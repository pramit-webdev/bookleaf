CONTEXT

BookLeaf Publishing is a self-publishing company that processes 1,200+ books monthly across a catalog
of 22,000+ titles. We work with thousands of authors across India and the US.

One of our biggest operational challenges is author communication at scale. Authors frequently reach out
with questions about their royalties, book status, ISBN issues, and printing quality. Currently, these
queries are handled manually, leading to delayed responses, inconsistent communication, and a growing
backlog—especially during high-volume periods.

Your assignment is to build a solution to this problem.


THE ASSIGNMENT

Build an Author Support & Communication Portal — a web application with two interfaces (author-
facing and admin-facing) powered by AI-assisted response generation. The system should help
BookLeaf’s team handle author support queries faster, more consistently, and with less manual effort.

  ⚠ Important: We are not looking for pixel-perfect UI. We are evaluating your architecture, code quality, AI
  integration, and how you think about real-world product problems. A clean, functional interface is sufficient.




DETAILED REQUIREMENTS

A. Author Portal (Author-Facing)
A simple dashboard where an author can log in and interact with their BookLeaf account.

 •   Authentication: Email-based login. No need for OTP or social login—simple email/password is
     fine. Each author should only see their own data.
 •   My Books: A page showing the author’s published books with key details: title, ISBN, genre,
     publication date, status, MRP, total copies sold, total royalty earned, royalty paid, and royalty
     pending.
 •   Submit a Support Query: A form where the author can raise a ticket. The form should include: a
     dropdown to select which book the query is about (or “General / Account Level”), a subject line, a
     detailed description field, and optional file attachment support (UI only—actual upload is bonus, not
     required).
 •   My Tickets: A page showing all tickets the author has submitted, with current status (Open, In
     Progress, Resolved, Closed), the original query, and any responses from the admin team. Tickets
     should update in real time or near-real time—if an admin responds, the author should see it without
     refreshing the page.

B. Admin Portal (Admin-Facing)
An internal dashboard for the BookLeaf operations team to manage incoming support queries.
  •   Ticket Queue: A list of all incoming tickets across all authors, filterable by status, category, priority,
      and date. The most urgent and oldest unresolved tickets should be easy to spot.
  •   AI-Powered Auto-Classification: When a new ticket comes in, the system should automatically
      classify it into one of these categories using an AI model: Royalty & Payments, ISBN & Metadata
      Issues, Printing & Quality, Distribution & Availability, Book Status & Production Updates, or General
      Inquiry. The admin should be able to override the AI’s classification if it’s wrong.
  •   AI-Generated Priority Score: Each ticket should get an automatic priority score (Critical / High /
      Medium / Low) based on the content. For example, a “I haven’t received any royalty for 6 months”
      should be flagged higher than “Can I update my author bio?” The admin should be able to override
      this too.
  •   AI-Drafted Response: When an admin opens a ticket, the system should present an AI-generated
      draft response based on the query content AND the BookLeaf Knowledge Base provided below.
      The admin should be able to edit the draft before sending. The sent response should appear on the
      author’s ticket view.
  •   Ticket Management: Admins should be able to update ticket status, add internal notes (not visible
      to the author), and assign tickets to themselves.

C. API Layer
The frontend and backend should communicate through a clean REST API. We expect:

  •   Properly structured endpoints with consistent naming conventions.
  •   Authentication middleware protecting all routes.
  •   Role-based access control: authors can only access their own data; admins can access everything.
  •   Input validation and meaningful error responses (not generic 500s).
  •   API documentation—either a Swagger/OpenAPI spec, a Postman collection, or a well-structured
      README section explaining every endpoint.

D. AI Integration Details
This is a critical part of the evaluation. We want to see how you think about integrating AI into a
production application, not just that you can make an API call.

  1. Prompt Engineering: Your prompts should be well-structured, use the Knowledge Base context
     we’ve provided, and produce responses that sound like a real BookLeaf support representative—
     not generic AI fluff.
  2. API Key Security: API keys must be stored in environment variables, never hardcoded or exposed
     to the frontend. We will check for this.
  3. Error Handling: What happens when the AI API is down or rate-limited? The system should
     degrade gracefully—tickets should still be created, and admins should be able to write responses
     manually.
  4. Cost Awareness: Are you sending the entire ticket history + knowledge base on every call, or are
     you being smart about token usage? We want to see evidence that you’ve thought about this.
5. Model Choice: You may use OpenAI, Anthropic, or any other LLM API you prefer. Justify your
   choice briefly in your documentation.
BOOKLEAF KNOWLEDGE BASE

Provide the following information to your AI system as context for generating responses. This is how
BookLeaf actually operates—your AI-drafted responses should reflect these policies accurately.

Company Overview
 •   BookLeaf Publishing is a self-publishing company operating in India and the US.
 •   We offer publishing packages: Standard Free (no upfront cost) and Bestseller Breakthrough
     (premium, paid package with marketing and distribution add-ons).
 •   We handle cover design, typesetting, ISBN assignment, printing, distribution, and royalty
     management for our authors.
 •   Our in-house printing facility and warehouse are located in Delhi. We also work with print partners
     including Repro India and Epitome Books.

Royalty Policy
 •   BookLeaf follows an 80/20 royalty split: 80% of the net profit per book goes to the author, 20% to
     BookLeaf.
 •   Net profit = MRP minus printing cost, platform commission (Amazon/Flipkart), and shipping charges.
 •   Royalties are calculated quarterly and paid within 45 days of the quarter ending.
 •   Minimum payout threshold: ₹1,000. If accumulated royalties are below this, they roll over to the next
     quarter.
 •   Payouts are made via bank transfer to the account linked in the author’s dashboard.
 •   Authors can view a detailed royalty breakdown in their dashboard, showing sales figures per
     platform.

ISBN Policy
 •   Every book published through BookLeaf receives a unique ISBN assigned by BookLeaf.
 •   ISBNs are registered under BookLeaf’s publisher imprint. If an author wants an ISBN under their
     own imprint, they need to obtain it independently.
 •   If an author reports an ISBN error (duplicate, wrong book linked), it is treated as a high-priority issue
     and escalated to the production team.

Printing & Quality
 •   In-house printing handles most orders. Overflow or specific format requirements go to Repro India
     or Epitome Books.
 •   Standard print turnaround: 5–7 business days from order confirmation.
 •   If an author reports a quality issue (misprints, binding defects, color inconsistency), BookLeaf
     arranges a free reprint after verification. The author may need to share photos of the defective copy.

Distribution & Availability
 •   Books are listed on Amazon India, Flipkart, Amazon US, Amazon UK, and the BookLeaf Store.
 •   New listings typically go live within 7–10 business days after publication is complete.
 •   If a book is showing as unavailable on a platform, it usually indicates a stock sync issue—
     BookLeaf’s team can trigger a re-sync within 24–48 hours.

Production Stages
 •   A book goes through the following stages: Manuscript Received → Editing (if opted) → Cover
     Design → Typesetting → Proofreading → ISBN Assignment → Printing → Distribution Setup →
     Published & Live.
 •   Authors are updated at each stage via email. Delays typically happen at Cover Design (waiting for
     author approval) and Proofreading (revision rounds).

Communication Tone Guidelines
This is how BookLeaf communicates with authors. The AI-generated responses should match this tone:

 •   Always empathetic and professional. Authors are our partners, not customers to be managed.
 •   Acknowledge the author’s concern before jumping to solutions.
 •   Be specific: include actual numbers, dates, and statuses wherever possible rather than vague
     reassurances.
 •   If something is BookLeaf’s fault (delayed royalties, ISBN error), own it directly. No corporate
     deflection.
 •   If the issue requires escalation or investigation, give a clear timeline (“Our team will look into this
     and get back to you within 48 hours”) rather than open-ended promises.
 •   Always end with a clear next step for the author and/or the BookLeaf team.

TECHNOLOGY GUIDELINES

  •   Frontend: React, Next.js, Vue, or any modern JavaScript framework. Your choice—pick what
      you’re strongest in.
  •   Backend: Node.js (Express/Fastify), Python (FastAPI/Django), or any backend framework. Again,
      your choice.
  •   Database: PostgreSQL, MongoDB, or any database you prefer. We care about your schema
      design, not which database you use.
  •   AI/LLM: OpenAI, Anthropic, or any LLM API. Use your own API key for development. If cost is a
      concern, OpenAI’s gpt-4o-mini or Anthropic’s Claude Haiku are inexpensive options that work well
      for this use case.
  •   Real-Time: WebSockets, Server-Sent Events, or polling—your choice. The ticket updates just need
      to feel responsive.
  •   Deployment: Deploy the application somewhere accessible (Vercel, Railway, Render, a VPS—
      anything). We need a live URL to test.
